Teaching a Language Model to Invoke Different Versions of Itself

March 2026

Abstract We investigate whether a frozen language model can learn to invoke specialized versions of itself through learned weight perturbations. Starting from a hypothesis that forward-inverse dynamics models could navigate the weight perturbation manifold, we iteratively discover that: (1) inverse dynamics fails completely on random LoRA bases; (2) a rigorous ablation proves random perturbations provide zero benefit beyond rejection sampling (p=0.87); (3) the fix is to learn task-specialized LoRA modes via fine-tuning, then train a GRPO policy to select among them. The final system achieves +0.050 reward improvement over the unperturbed baseline (p<0.0001) on entity-constrained story generation, with the GRPO policy outperforming even the hand-designed oracle mapping. The journey from negative result to working system is the main contribution.

1. The Hypothesis and Its Failure

We start with an idea from robotics: if you can learn the forward dynamics of a system (given state + action, predict next state) and its inverse (given current + desired state, solve for action), you can plan trajectories through the system's state space. We attempt to apply this to the "weight perturbation manifold" of a frozen language model.

The setup: take a frozen 68M-parameter TinyStories model (GPT-Neo, 4 layers, 768 hidden). Define K=32 random rank-1 LoRA directions as an "action space." Collect 20,000 state transitions under random perturbations. Train forward and inverse dynamics MLPs. Use them to plan perturbations that steer the model's "cognitive state" toward desired configurations.

1.1 What Worked (v6.0)

The forward model learns real structure: 0.97 cosine similarity on state deltas, meaning it correctly predicts which direction the model's internal state will shift under a given perturbation. The per-sample LoRA hook system processes 205 transitions/second on a single RTX 3090. The entire pipeline runs in 103 seconds.

The mapping from 32-dimensional actions to 64-dimensional state observations is many-to-one: many different actions produce indistinguishable state changes.

1.3 Iteration Attempts (v6.1–v6.3)

We attempted several fixes: dropping the inverse model in favor of forward-model search (v6.1), increasing perturbation strength from α=0.06 to 0.20 (v6.1), reducing K from 32 to 8 modes (v6.1), replacing hand-crafted state features with random projections (v6.1), and training a GRPO policy to select among dream modes (v6.1). None produced significant improvement. The GRPO policy either collapsed to "never dream" (v6.2) or "always dream with the same mode" (v6.1), because the underlying perturbations didn't create meaningfully different text distributions.

We then tried best-of-N action search (v6.2–v6.3), which appeared to show massive improvement: best-of-256 actions scored 0.90 vs baseline 0.57 (+0.33). But the robustness check (v6.3) revealed that "best" actions don't transfer across random seeds: Robust Δ = -0.0002, only 40% improved. This demanded a definitive ablation.

2. The Ablation That Changed Everything

The critical experiment: if we generate N=128 samples with no perturbation and pick the best (pure rejection sampling), does it perform as well as generating N=128 samples with random LoRA perturbations and picking the best?

Random LoRA perturbations provide ZERO benefit. The difference between best-of-N with vs without perturbation is +0.0015 (paired t-test: t=0.161, p=0.87). All of the apparent improvement was rejection sampling: with 128 tries, you're likely to hit a good generation by chance. The best strategy is simply generating many completions with varying base prefixes (best-of-128 fresh base: 0.93). The LoRA perturbation manifold, initialized with random bases, does not expand the reachable output space in any meaningful direction.

This is a fundamental negative result: random perturbation directions in weight space are not semantically meaningful. Perturbing a random direction in a neural network's weight space is like pushing a robot's joints in a random direction—it moves, but not toward anything useful. To get directional control, the perturbation directions must be learned.

3. The Fix: Learned LoRA Modes

The ablation made the path forward clear: instead of random LoRA bases, learn them. We define K=4 LoRA modes, each specialized for a category of content:

ModeCategoryExample Entities
0Animalscat, dog, bird, fish, rabbit, bear, mouse, fox, turtle, frog
1Charactersprincess, knight, wizard, dragon, fairy, queen, king, pirate
2Placesriver, mountain, castle, garden, forest, cave
3Objectstree, flower, boat, lamp, key, crown, sword, shield, map, book

3.1 Training Pipeline

  1. Generate training data: Generate 2,000 stories from the base model. Filter by which entity categories appear in each story. This produces ~600–900 stories per category.
  2. Fine-tune each mode's LoRA: For each mode k, install its rank-8 LoRA perturbation via forward hooks (differentiable), run the standard language modeling loss on the corresponding entity-category stories, and update only the LoRA A/B parameters. 40 epochs, lr=5e-4. The base model stays frozen.

Each mode has its own LoRA A and B matrices (688K total learnable parameters across all 4 modes, applied to all 8 target layers). The fine-tuning teaches each mode to increase the probability of generating text mentioning its target entity category.

4. GRPO: Learning When to Dream

With working LoRA modes, we need a policy that decides which modes to activate for a given task. The task: generate a story including 3 specific entities (e.g., "cat, castle, river"). The reward is a composite of entity coverage (50%), n-gram diversity (20%), vocabulary diversity (10%), length (10%), and coherence (10%).

4.1 Architecture (Critical Detail)

An earlier attempt (v6.4) used softmax over K mode logits, forcing modes to compete. This was catastrophic: with softmax, each mode gets at most 1/K of the activation weight, making the effective perturbation too weak. The fix:

Independent sigmoid activations. Each mode has its own activation probability in [0,1], decided independently. The policy can activate any combination of modes simultaneously at full strength. This allows the model to invoke multiple "versions of itself" at once.

class ModePolicy(nn.Module):
    def forward(self, h):
        # h: last-token hidden state from the frozen base model
        return torch.sigmoid(self.net(h))  # (B, K) each in [0,1]
        # NOT softmax! Modes are independent, not competing.

4.2 GRPO Training

800 prompts, group_size=6. For each prompt, sample 6 binary activation vectors from the policy's Bernoulli distribution, generate with those modes active, score the reward, compute REINFORCE gradients with a group-mean baseline and entropy regularization.

5.2 Hierarchy of Methods

The results establish a clear ordering:

  1. GRPO policy (0.617): Best. Learned which modes help and which don't.
  2. Oracle mapping (0.590): Second. Knows which entity maps to which mode, but activates the weak characters mode.
  3. All modes on (0.584): Third. Blunt instrument—activating everything helps, but includes the unhelpful characters mode.
  4. Random mode (0.578): Fourth. Not significant (p=0.20). A single random learned mode is no better than base.
  5. Base model (0.568): Baseline.

The gap between "random learned mode" (not significant) and "GRPO policy" (p<0.0001) demonstrates that mode selection matters, not just mode existence. Having specialized model versions is necessary but not sufficient—you need a policy that knows when to invoke each one.

6. Engineering: Per-Sample LoRA via Forward Hooks

The key engineering contribution enabling this work: per-sample weight perturbation in a batched forward pass.

6.1 The Problem

Standard LoRA applies the same perturbation ΔW to all samples in a batch. To test different perturbations for different samples (essential for both data collection and GRPO), you'd need sequential processing: modify weights → forward one sample → revert → repeat. This serializes GPU computation.

6.2 The Solution

For a linear layer y = Wx + b, the perturbed output y' = (W + ΔW)x + b = Wx + b + ΔWx = y + ΔWx. We compute the per-sample correction ΔWx via a forward hook:

def hook(module, inp, out):
    x = inp[0]                                           # (batch, seq, in)
    per_dw = einsum("bk,koi->boi", actions, dw_basis)    # per-sample ΔW
    correction = einsum("boi,bsi->bso", per_dw, x)       # per-sample correction
    return out + alpha * correction

Two batched einsum operations, mathematically equivalent to a separate weight matrix per sample. Integrates with HuggingFace .generate() and KV caching. On an RTX 3090 with the 68M-param model, this achieves 209 transitions/second at batch_size=64.

6.3 For Training (Differentiable)

The same hook architecture enables differentiable LoRA fine-tuning. Since the LoRA A and B matrices are nn.Parameters and the hook's computation (matrix multiply + addition) is differentiable, calling loss.backward() automatically computes gradients for the LoRA parameters. The base model's parameters have requires_grad=False, so no base model gradients are computed.

7. Discussion

7.1 What This Demonstrates

The core contribution: a frozen language model can learn to invoke specialized versions of itself, and a GRPO-trained policy can learn when each version is appropriate. The "versions" are LoRA perturbation modes, each trained on a content category. The policy observes the task (via the base model's hidden state) and decides which modes to activate, outperforming both the base model and a hand-designed oracle.

7.2 Why Random Bases Fail

The ablation result (Section 2) has a simple explanation: a random direction in weight space does not correspond to any semantic direction in output space. Neural network loss landscapes are highly structured, and meaningful behavioral changes (e.g., "generate more animal words") live in very specific directions. Random perturbations produce noise in output space, not meaningful variation. Fine-tuning discovers these specific directions.

7.3 The Dynamics Model was Wrong

Our original hypothesis was that forward-inverse dynamics could navigate the perturbation manifold. The forward model worked (0.97 cosine on direction), but the inverse model was doomed from the start: the mapping from actions to states is many-to-one with random bases, and even with learned bases, the action-to-output mapping is non-injective. The fix was to skip dynamics modeling entirely and use direct reward optimization (GRPO).

7.4 Independent Activations vs Softmax

A subtle but critical architectural detail: the policy must use independent sigmoid activations, not softmax. With softmax, modes compete for activation weight, and the effective per-mode perturbation is 1/K (too weak at K=4). With independent sigmoids, each mode can be fully activated simultaneously, allowing the model to invoke multiple versions of itself at once. The GRPO policy's convergence pattern (Section 4.3) would be impossible with softmax.

7.5 Limitations

  • Scale: Tested on a 68M-param model only. Larger models may need different α, rank, and more modes.
  • Task: Entity inclusion is a relatively simple task. More complex steering goals (tone, style, reasoning quality) may be harder.
  • Mode categories are hand-defined. An automatic clustering of training data could remove this design choice.
  • The characters mode is weak (+10%). This may be because TinyStories already frequently generates character-related content (10% base rate is low compared to animals at 26%).
  • The GRPO policy is task-agnostic. It learns "always activate modes 0, 2, 3" regardless of which specific entities are required. A more capable policy would condition on the actual task content.

7.6 Future Directions

Task-conditioned mode selection. The current policy ignores which entities the task requires. Incorporating entity information (e.g., the entity-to-mode mapping) as an input feature would allow truly adaptive mode selection.

End-to-end LoRA base learning. Instead of fine-tuning modes on pre-categorized data, learn the LoRA bases directly from reward using evolutionary strategies or gradient estimation. This would discover modes aligned with the actual task objective, not proxy categories.

Hierarchical modes. Beyond 4 coarse categories, a hierarchy of modes (coarse → fine) could provide more precise steering: "animals" → "aquatic animals" → "fish".

Inference-time mode search. Combine learned modes with test-time search: use the policy as a warm start, then refine with a few CEM iterations for challenging tasks.

Appendix D: Ablation Details

Full ablation experiment: 100 tasks, 128 candidates per condition. All conditions use the same set of tasks for paired comparison.

ConditionMean RewardStdInterpretation
Single sample0.56700.119Baseline expected value
Mean-of-128 (no pert)0.56940.085Average quality, no perturbation
Mean-of-128 (with pert)0.56830.086Average quality, random perturbation
Best-of-128 (no pert, same base)0.87410.095Rejection sampling ceiling

Best-of-128 (no pert, fresh base)0.93380.062Varying base prefix > LoRA

Key insight: the mean-of-128 with perturbation (0.5683) is actually slightly lower than without (0.5694), with Δ = -0.0011 (p=0.39). Random LoRA perturbation at α=0.5 very slightly degrades mean quality while not expanding the distribution's upper tail. The improvement from selection (0.567 → 0.874) comes entirely from the stochastic decoder's natural variation.

The fresh-base condition (0.934) outperforms same-base (0.874) because varying the base prefix creates more diverse semantic starting points than varying LoRA perturbations. This is the strongest evidence that random LoRA directions are semantically inert.
